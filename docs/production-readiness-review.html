A fronteira de compatibilidade cognitiva: Paganini é internamente sofisticado — tem router bayesiano, memory episódica, skill injection, graph RAG. Mas toda essa riqueza é opaca para quem opera. É como ter um motor V12 sem painel de instrumentos. A observabilidade precisa ser isomórfica à arquitetura interna: cada camada cognitiva (roteamento, retrieval, guardrail, memória, metacognição) precisa de sua própria superfície de observação — traces, métricas, dashboards.

O gap metacognitivo: MetaClaw faz auto-melhoria, mas o sistema não observa a si mesmo melhorando. Sem métricas de before/after de skill injection, sem A/B testing de prompts, sem decay tracking de memória. A recursividade neural exige que a camada de observação alimente a camada de aprendizado — o sistema vê que melhorou, e isso informa o que tentar next.

Roadmap para Primeira Entrega

Sprint 1 — Foundation (Semanas 1-2)

Metering Service

Token counter em Moltis proxy → cost calculator → persistence (Supabase) → billing API. Sem isso, não há modelo de negócio.

Auth & RBAC

JWT tokens per-tenant. Roles: admin, operator, viewer. API key per-tenant com rate limits. RLS no pgvector por tenant_id.

PII Redaction Gate

Guardrail pre-LLM que redacta CPF, CNPJ, nomes, contas bancárias antes de enviar ao provider. Non-negotiable para finance.

Sprint 2 — Quality Gate (Semanas 3-4)

Eval Dataset (FIDC)

50+ Q&A pairs com expected sources. Golden dataset para AutoResearch e CI quality gate. Rodar primeiro ciclo de AutoResearch.

Hallucination Detector

Post-generation gate: NLI model verifica se resposta é entailed pelo context retrieved. Bloqueia respostas não-grounded.

HITL Workflow

Operações críticas (cessão, relatório regulatório) entram em fila de aprovação. Slack/Dashboard notifica. Humano aprova/rejeita. Audit trail completo.

CI/CD Completo

Docker build → push GHCR → deploy staging → smoke test → manual gate → prod. Tests rodam em CI. Coverage enforced.

Sprint 3 — Observability (Semanas 5-6)

OpenTelemetry Integration

Spans em cada camada: Router decision → Agent dispatch → RAG retrieval → Guardrail check → LLM call → Response. Cada span com tenant_id, agent_id, model_used, tokens, cost.

Operator Dashboard

Não o dashboard do usuário — o dashboard do operador. Token usage by tenant. Guardrail block rate. RAG quality trending. Agent performance. Cost projection.

Feedback Loop

👍/👎 no chat → MetaClaw skill scoring → AutoTune EMA. O sistema começa a aprender do uso real.

Corpus Ingestion Pipeline

Cliente faz upload de documentos → chunking → embedding → indexação → RAG disponível. Self-service via dashboard ou CLI.

Avaliação Final

O paradoxo do Paganini

O sistema é arquiteturalmente sofisticado (router bayesiano, graph RAG, skill evolution, multi-agent orchestration, guardrail pipeline) mas operacionalmente imaturo (sem metering, sem auth multi-tenant, sem eval datasets, sem HITL, sem OTel, sem CD).

É um V12 montado numa garagem — potência bruta existe, mas faltam freios (guardrails testados), painel (observabilidade), chave (auth), e hodômetro (metering).

12 Blockers P0

  • Metering / cost attribution
  • Rate limiting per tenant
  • Auth RBAC multi-tenant
  • Data isolation (RLS)
  • PII redaction pre-LLM
  • Secret management (vault)
  • Hallucination detection
  • Eval datasets
  • HITL workflow
  • Docker image build + registry
  • CI tests rodando
  • Corpus self-service ingestion

A Tese de Entrega

Primeira entrega viável em ~6 semanas com foco em:

  • Single-tenant managed deploy (nós operamos, cliente usa)
  • Slack como canal principal (canal #fund-X por fundo)
  • FIDC pack only (multi-domain vem depois)
  • HITL para operações críticas
  • Metering para billing transparente

Simplificar scope: managed single-tenant com Slack elimina 4 dos 12 blockers (multi-tenant auth, RLS, dashboard user testing, corpus self-service).
